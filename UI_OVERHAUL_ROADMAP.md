# UI Overhaul Roadmap

This document captures the remaining UI modernization work after the landing page, auth pages, and dashboard redesign. Use it as the planning source for future UI work.

## Current State

Already modernized:

- Landing/title screen at `/`
- Login at `/login`
- Signup at `/signup`
- Dashboard at `/dashboard`

Remaining major surfaces:

- Local play at `/play`
- Online play at `/play/online`
- Legacy in-game account menu mounted by public scripts
- In-game notifications/toasts
- Game room/lobby UI rendered by legacy scripts
- UI Lab cleanup/polish at `/ui-lab`, if desired

## Design Direction

Keep the established visual language:

- TheGridCN-inspired HUD/Tron command-center styling
- Orbitron display typography
- Rajdhani body typography
- Transparent panels, cyan borders, reticles, radar, terminals, and scan overlays
- Minimal solid fills; prefer glassy cards, outlines, glows, and grid-backed surfaces
- Desktop-first layouts with responsive fallback for smaller screens

## Guiding Principles

- Preserve game functionality before visual polish.
- Replace legacy UI incrementally, not all at once.
- Keep server/game logic unchanged unless UI work exposes a real bug.
- Prefer shared React components over duplicated HTML in public scripts.
- Avoid browser-native alerts/prompts/popups; use TheGridCN-styled dialogs/toasts.
- Keep all interactive states visible: loading, disabled, ready, error, success, disconnected.

## Phase 1: Local Play Page

Route: `/play`

Current state:

- Uses `LegacyGameShell`.
- Mounts legacy `/js/local.js` into `#game-root`.
- Styling comes mostly from `/game.css` and public DOM generation.

Goal:

- Make local play feel like a polished arena screen instead of a legacy embedded page.

Recommended work:

- Create a modern React shell around the game area.
- Add a persistent HUD header with:
  - game mode: `Local Play`
  - player count
  - current round/status
  - return/dashboard controls
- Restyle the local game setup screen with TheGridCN cards and controls.
- Convert key player states into clear HUD panels:
  - waiting
  - ready
  - false start
  - reaction time
  - round winner
  - match winner
- Replace legacy notification styling with app-level HUD notifications.

Acceptance criteria:

- Local play still supports existing 2-4 player flow.
- Existing keyboard behavior remains intact.
- Page visually matches the dashboard/title screen direction.
- No raw browser popups or default form styling remain.

## Phase 2: Online Play Page

Route: `/play/online`

Current state:

- Uses `LegacyGameShell`.
- Requires authentication server-side.
- Mounts legacy `/js/remote.js` and Socket.IO client.

Goal:

- Modernize online rooms/lobby/gameplay while preserving real-time behavior.

Recommended work:

- Create a modern online play shell with sections for:
  - create room
  - join room
  - lobby members
  - ready states
  - host controls
  - chat/events feed
  - active round
- Convert room code display into a prominent HUD component.
- Replace legacy room/lobby panels with TheGridCN cards.
- Add connection status indicators:
  - connected
  - reconnecting
  - disconnected
  - kicked/blacklisted
- Add clear host-only visual treatment for kick/start controls.
- Keep chat/event feed readable and compact.

Acceptance criteria:

- Online room create/join flow still works.
- Ready/start/round scoring still matches existing socket behavior.
- Host reclaim/kick/blacklist flows remain functional.
- Socket connection states are visible to the user.

## Phase 3: Legacy Script Reduction

Current state:

- Gameplay UI is still generated by scripts in `public/js`.
- React shell mainly loads and coordinates those scripts.

Goal:

- Gradually move UI rendering into React while keeping core game logic stable.

Recommended strategy:

- First extract read-only render pieces into React components.
- Keep event wiring in existing scripts until components are stable.
- Move state ownership only after the UI is visually complete.
- Avoid a full rewrite unless there is enough time to test online gameplay thoroughly.

Candidate React components:

- `GameHud`
- `PlayerPanel`
- `RoundStatusPanel`
- `ReactionResultCard`
- `RoomCodePanel`
- `LobbyRoster`
- `ChatFeed`
- `ReadyControlPanel`
- `GameToast`

Acceptance criteria:

- Public scripts shrink over time instead of growing.
- React components own presentation.
- Game logic remains covered by current tests.

## Phase 4: Account Menu And Notifications

Current state:

- `accountMenu.js` mounts a legacy account menu in game pages.
- `pageNotifications.js` provides legacy notification behavior.

Goal:

- Replace legacy account/notification UI with React/TheGridCN equivalents.

Recommended work:

- Reuse the non-dropdown auth badge pattern from dashboard.
- Add profile/settings entry points consistently across pages.
- Replace page notifications with a shared TheGridCN toast system.
- Use consistent message categories:
  - success
  - warning
  - error
  - system
  - game event

Acceptance criteria:

- Notifications look consistent on auth, dashboard, local play, and online play.
- Account controls no longer depend on legacy DOM mutation.

## Phase 5: UI Lab Cleanup

Route: `/ui-lab`

Current state:

- Useful for theme/intensity/atmosphere testing.
- Large and likely more developer-facing than player-facing.

Goal:

- Decide whether UI Lab remains a dev tool or becomes a player-facing settings page.

Options:

- Keep as developer-only visual sandbox.
- Convert into player-facing appearance settings.
- Split into `/ui-lab` for dev and dashboard profile/settings for players.

Recommended direction:

- Keep `/ui-lab` as a developer sandbox for now.
- Move only user-facing theme controls into the future dashboard Profile/Settings section.

Acceptance criteria:

- UI Lab does not block game modernization.
- Any player-facing controls are accessible from dashboard.

## Suggested Implementation Order

1. Modernize `/play` local shell first.
2. Modernize `/play/online` lobby next.
3. Modernize active online gameplay states.
4. Replace legacy account menu and notifications.
5. Move reusable player settings into dashboard Profile.
6. Decide final role of `/ui-lab`.

## Testing Checklist

Run after each phase:

- `npm test`
- `npm run build`

Manual checks:

- Landing page loads cleanly.
- Login/signup errors display inline.
- Dashboard navigation and scroll-spy work.
- Local play can complete a match.
- Online room can be created and joined.
- Host can start game.
- Players can ready/unready.
- Disconnect/kick/error states are visible.

## Risks

- Rewriting public game scripts too quickly could break socket/game behavior.
- Visual layering can conflict with legacy CSS in `/game.css`.
- Online play needs manual multi-client testing beyond unit tests.
- Excessive decorative overlays can hurt readability during gameplay.

## Definition Of Done

The UI overhaul is complete when:

- All main routes share the same visual language.
- Legacy-looking form/card styles are gone.
- Local and online gameplay screens feel like part of the same app as landing/dashboard.
- All browser-native popups are replaced with styled UI.
- Tests and production build pass.
- Manual local and online gameplay flows are verified.
